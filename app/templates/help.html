Lab Manager

Панель управляет пользовательскими приложениями через systemd и пробрасывает их наружу через отдельные TCP-порты с авторизацией. Приложение ничего не знает о панели — оно видит себя на корне /.

Как это работает

1. Папка с кодом
Код живёт в /root/apps/<имя>. Туда можно клонировать из git, загрузить архив или просто положить папку.
2. systemd-юнит
Lab Manager создаёт юнит labapp-<имя>.service с Restart=always, пробрасывает PORT/HOST и запускает процесс на 127.0.0.1.
3. Проброс порта
Панель слушает внешний порт на 0.0.0.0, проверяет cookie авторизации и сырым TCP-сплайсом перенаправляет трафик на 127.0.0.1:<PORT>.

Три способа добавить приложение

На главной странице → форма «Клонировать из Git». Укажите имя и URL репозитория (https://... или git@...). Панель сделает git clone --depth 1, а затем автоматически создаст venv и поставит зависимости из requirements.txt, если он есть.

Поддерживаются .zip, .tar.gz, .tgz, .tar. Панель распакует архив в /root/apps/<имя> (с защитой от Zip Slip), схлопнет один корневой каталог, если он есть, и так же автоматически поднимет venv.

Положите папку в /root/apps/ любым удобным способом (rsync, scp, cp). Она сразу появится на дашборде. Нажмите кнопку «Настроить окружение» для создания venv.

Автоматическая настройка venv

После git clone, загрузки архива или git pull панель выполняет:

  1. Проверяет, есть ли признаки Python-приложения (app.py, main.py, wsgi.py или requirements.txt).
  2. Если venv отсутствует или сломан — создаёт заново: python3 -m venv venv.
  3. Если есть requirements.txt — venv/bin/pip install -r requirements.txt.

Кнопка «Настроить окружение» на странице приложения позволяет запустить пересборку вручную в любое время.

Проброс портов

Каждому приложению можно назначить внешний порт (1024–65535). Lab Manager откроет этот порт на 0.0.0.0 и будет действовать как прозрачный TCP-прокси с авторизацией:

  1. Пользователь обращается к http://<host>:<порт>/.
  2. Lab Manager читает HTTP-заголовки и извлекает cookie labmgr_session.
  3. Если сессия валидна и у пользователя есть доступ к приложению — трафик сырым TCP-splice перенаправляется на 127.0.0.1:<PORT>.
  4. Если не авторизован — 302 на страницу логина панели.
Почему это работает:
  • Приложение на корне / — никаких префиксов, никаких X-Script-Name. Любые URL (абсолютные, относительные, JS fetch) работают как есть.
  • Любые протоколы — HTTP/1.1, WebSocket, SSE, chunked encoding.
  • Single Sign-On — cookie labmgr_session по RFC 6265 не привязана к порту, поэтому браузер автоматически отправляет её на любой порт того же хоста. Войти в панель достаточно один раз.
  • Изоляция — cookies приложений не пересекаются (разные порты), а cookie панели (labmgr_session) имеет уникальное имя.
Настройка:

Страница приложения → карточка «Внешний порт» → ввести номер → «Включить». Внешний порт не должен совпадать с внутренним портом приложения и портом панели (по умолчанию 80).

Требования к коду приложения

Благодаря прозрачному пробросу портов требования минимальны:

Обязательно
  • Слушать host и port из env: 
    import os
host = os.environ.get("HOST", "127.0.0.1")
port = int(os.environ["PORT"])
app.run(host=host, port=port)
  • Отключить reloader: debug=False, use_reloader=False — иначе systemd не сможет штатно остановить процесс.
Рекомендуется
  • Абсолютные пути к файлам: BASE_DIR = os.path.dirname(os.path.abspath(__file__))
Нельзя
  • Хардкодить порт.
  • Биндить 0.0.0.0 — только 127.0.0.1 (через переменную HOST).
  • Переопределять PORT, HOST, PYTHONUNBUFFERED в env-переменных.
Никаких ограничений на URL в HTML/JS: абсолютные пути (fetch('/api/x')), относительные, любые фреймворки — всё работает, потому что приложение видит себя на корне / собственного порта.

Минимальное Flask-приложение

# app.py
import os
from flask import Flask, jsonify

BASE_DIR = os.path.dirname(os.path.abspath(__file__))
app = Flask(__name__)

@app.route("/")
def index():
    return "<h1>Hello from Lab Manager!</h1>"

@app.route("/api/ping")
def ping():
    return jsonify(ok=True)

if __name__ == "__main__":
    app.run(
        host=os.environ.get("HOST", "127.0.0.1"),
        port=int(os.environ["PORT"]),
        debug=False,
        use_reloader=False,
    )

# requirements.txt

Жизненный цикл сервиса

  1. Создание: «Создать сервис» → генерируется unit-файл labapp-<name>.service → daemon-reload → enable → start.
  2. Настройка внешнего порта: «Включить» → TCP-форвардер поднимается на указанном порту. Значение сохраняется в БД и восстанавливается при перезапуске панели.
  3. Редактирование: меняется unit-файл, если сервис был активен — автоматический restart.
  4. Старт/стоп/рестарт: прямые команды systemctl.
  5. Удаление:
    1. Останавливается TCP-форвардер (освобождает внешний порт).
    2. systemctl stop (graceful, до 10 сек).
    3. Ожидание перехода в inactive.
    4. Если процесс ещё жив — SIGKILL по всему дереву через psutil.
    5. Safety-net: kill всех, кто висит на назначенном порту.
    6. disable, reset-failed, удаление unit-файла.
  6. Диагностика: запускает приложение на 5 секунд в изолированном процессе и собирает stderr.

Пользователи и права доступа

  • Администратор — видит полный дашборд, создаёт/правит/удаляет приложения, управляет пользователями, назначает внешние порты.
  • Обычный пользователь — видит только те приложения, на которые ему выдан доступ. Может открывать приложения через назначенные внешние порты. Никаких кнопок управления.
  • Первый админ создаётся из переменной ADMIN_PASSWORD в .env.
  • Удалить последнего администратора или самого себя нельзя.
  • При удалении приложения все связанные права автоматически очищаются.

Что делать, если не работает

СимптомГде смотретьЧто делать
Сервис сразу падает «Диагностика» на странице приложения Читайте stderr. Обычно ModuleNotFoundError — нажмите «Настроить окружение».
Не открывается на внешнем порту Проверьте, что порт назначен и сервис запущен Включите внешний порт на странице приложения. Если порт занят — выберите другой.
302 на логин при каждом запросе — Не авторизованы. Зайдите в панель (:80), залогиньтесь — cookie подхватится автоматически на всех портах.
Изменения в коде не подхватываются — Нажмите «Restart». Restart=always перезапускает только при падении, не при изменении файлов.
«venv сломан» или Permission denied Flash-сообщение Кнопка «Настроить окружение» — панель снесёт и пересоберёт venv.
