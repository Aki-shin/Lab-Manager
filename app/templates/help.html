🛠 Lab Manager — как тут всё устроено

Панель управляет пользовательскими приложениями через systemd, раздаёт им порты из диапазона 5001–6000 и выставляет наружу по единому адресу через встроенный reverse-proxy. Всё под root, всё на systemd, никакой магии.

🔧 Как это работает

1. Папка с кодом
Код живёт в /root/apps/<имя>. Туда можно клонировать из git, загрузить архив или просто положить папку.
2. systemd-юнит
Lab Manager создаёт юнит labapp-<имя>.service с Restart=always, пробрасывает PORT/HOST и запускает процесс.
3. Reverse-proxy
Внешний запрос /proxy/<имя>/... панель пересылает на 127.0.0.1:<port> с корректными X-Forwarded-* заголовками.

➕ Три способа добавить приложение

На главной странице → форма «Клонировать из Git». Укажите имя и URL репозитория (https://... или git@...). Панель сделает git clone --depth 1, а затем автоматически создаст venv и поставит зависимости из requirements.txt, если он есть.

Поддерживаются .zip, .tar.gz, .tgz, .tar. Панель распакует архив в /root/apps/<имя> (с защитой от Zip Slip), схлопнет один корневой каталог, если он есть, и так же автоматически поднимет venv.

Положите папку в /root/apps/ любым удобным способом (rsync, scp, cp). Она сразу появится на дашборде. В этом случае нажмите на приложении кнопку «Настроить окружение», чтобы панель сама собрала venv.

⚙️ Автоматическая настройка venv

После git clone, загрузки архива или git pull панель выполняет:

  1. Проверяет, есть ли признаки Python-приложения (app.py, main.py, wsgi.py или requirements.txt).
  2. Если venv отсутствует или сломан — создаёт заново: python3 -m venv venv.
  3. Если есть requirements.txt — venv/bin/pip install -r requirements.txt.
  4. Результат показывается во flash-сообщении. Таймаут на установку — 10 минут.

Когда запускать вручную: после ручного обновления кода или при смене requirements.txt без git pull. Кнопка есть на странице приложения.

📋 Требования к коду приложения

✅ Обязательно
  • Слушать host/порт из env:
    host = "127.0.0.1"
    port = int(os.environ["PORT"])
  • Отключить reloader: debug=False, use_reloader=False. С reloader systemd получает «не тот» pid и не может штатно остановить процесс.
  • Абсолютные пути к файлам:
    BASE_DIR = os.path.dirname(os.path.abspath(__file__))
    WorkingDirectory у сервиса задан правильно, но всё равно — привычка полезная.
  • Относительные URL на фронте: fetch('api/x'), а не fetch('/api/x'). Приложение живёт под префиксом /proxy/<name>/.
❌ Нельзя
  • Хардкодить порт — приходит PORT от менеджера, диапазон 5001-6000.
  • Биндить 0.0.0.0 — приложение должно быть доступно только через reverse-proxy. Используйте 127.0.0.1.
  • Абсолютные URL на фронте: fetch('/api/x') уйдёт в корень панели, а не в приложение.
  • Переопределять PORT/HOST/PYTHONUNBUFFERED в пользовательских env-переменных — эти имена зарезервированы.

🔑 Переменные окружения

Lab Manager автоматически прокидывает три переменные, трогать которые нельзя:

ПеременнаяЗначениеЗачем
PORTсвободный порт 5001-6000приложение слушает на нём
HOST127.0.0.1никакого выхода наружу в обход прокси
PYTHONUNBUFFERED1логи идут в journalctl без задержек

Собственные переменные (DB_URL, API_KEY и т.д.) добавляются в форме «Создать/Редактировать сервис». Имя должно соответствовать ^[A-Z_][A-Z0-9_]*$.

🧪 Минимальное Flask-приложение

Закинь этот app.py + пустой requirements.txt с flask, и оно заведётся из коробки:

# app.py
import os
from flask import Flask, jsonify

BASE_DIR = os.path.dirname(os.path.abspath(__file__))
app = Flask(__name__)

@app.route("/")
def index():
    return "<h1>Hello from Lab Manager!</h1>"

@app.route("/api/ping")
def ping():
    return jsonify(ok=True)

if __name__ == "__main__":
    app.run(
        host=os.environ.get("HOST", "127.0.0.1"),
        port=int(os.environ["PORT"]),
        debug=False,
        use_reloader=False,
    )

# requirements.txt
flask>=3.0

🔀 Как работает reverse-proxy

Запрос GET /proxy/myapp/api/users?id=1 превращается в GET http://127.0.0.1:<PORT>/api/users?id=1 и ретранслируется обратно клиенту.

В заголовки приложения панель добавляет:

  • X-Script-Name: /proxy/myapp — если фреймворк умеет уважать (Flask + ProxyFix), он сам построит корректные URL.
  • X-Forwarded-For, X-Forwarded-Proto, X-Forwarded-Host.

Важно: если приложение отдаёт редирект на /login, панель переписывает Location на /proxy/myapp/login автоматически. Но HTML с захардкоженными <a href="/..."> переписать нельзя — поэтому используйте относительные URL.

♻️ Жизненный цикл сервиса

  1. Создание: кнопка «Создать сервис» → генерируется /etc/systemd/system/labapp-<name>.service → daemon-reload → enable → start.
  2. Редактирование: меняется unit-файл, если сервис был активен — выполняется restart.
  3. Старт/стоп/рестарт: прямые команды systemctl.
  4. Удаление:
    1. systemctl stop (graceful, до 10 сек).
    2. Ожидание перехода в inactive.
    3. Если процесс ещё жив — SIGTERM + SIGKILL по всему дереву через psutil.
    4. Safety-net: kill всех, кто ещё висит на назначенном порту.
    5. disable, reset-failed, удаление unit-файла, daemon-reload.
  5. Диагностика: кнопка «Диагностика» запускает приложение на 5 секунд в изолированном процессе и собирает stderr. Если упало сразу — получите трассировку без необходимости лезть в journalctl.

👥 Пользователи и права доступа

  • Администратор — видит полный дашборд, создаёт/правит/удаляет приложения, управляет пользователями.
  • Обычный пользователь — видит только те приложения, на которые ему выдан доступ. На странице «Мои приложения» он кликает «Открыть» и попадает в приложение через прокси. Никаких кнопок управления ему не показывается.
  • Первый админ создаётся из переменной ADMIN_PASSWORD в .env при первом старте.
  • Удалить последнего администратора или самого себя нельзя — панель не даст.
  • При удалении приложения все связанные права автоматически очищаются.

🩺 Что делать, если не работает

СимптомГде смотретьЧто делать
Сервис сразу падает «Диагностика» на странице приложения Читайте stderr — чаще всего это ModuleNotFoundError. Нажмите «Настроить окружение».
502 Bad Gateway в прокси Статус в панели + «Логи» Приложение не слушает на $PORT или забиндилось в 0.0.0.0 и падает из-за конфликта.
Относительные пути сломали фронт DevTools → Network Запросы уходят на /api/... вместо /proxy/name/api/... — уберите ведущий слэш.
Изменения в коде не подхватываются — Нажмите «Restart». Restart=always перезапускает только при падении, не при изменении файлов.
«venv сломан» или Permission denied Flash-сообщение Кнопка «Настроить окружение» — панель снесёт и пересоберёт venv.
После удаления процесс всё ещё висит ss -tlnp, ps aux С версии 2 это невозможно: удаление бьёт дерево процессов и освобождает порт. Если всё-таки — шлите баг.
